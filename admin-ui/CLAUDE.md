# CLAUDE.md — telegram_server 관리자 UI

이 문서는 `admin-ui/` 하위에서 작업하는 모든 Claude 세션의 운영 지침이다.
**루트 `CLAUDE.md`(R1~R8)는 그대로 상속**되며, 이 문서는 관리자 UI 특화 규칙과 Claude Fable 5 프롬프팅 가이드 기반 작업 원칙을 추가한다.

---

## 1. 프로젝트 컨텍스트 (왜 만드는가)

telegram_server의 관리 작업이 현재 3개 창구(curl로 /admin API, DB 직접 SQL, 봇 명령)에 흩어져 있고, 특히 **API 키 발급이 완전 수동**(해시 생성 → SQL INSERT)이다. 운영자가 브라우저에서 안전하게 관리할 수 있는 단일 UI를 제공한다.

핵심 기능 범위:
1. 앱 관리 — 등록/수정/비활성화, capability 부여·회수 (`/admin/apps` 래핑)
2. **API 키 발급/폐기** — secret 생성 → Argon2id 해시(`internal/auth.HashAPIKey`) → `app_keys` INSERT, 평문 1회 표시. 현재 시스템의 최대 gap
3. 사용자 등급 관리 (`/admin/users/{telegram_id}`)
4. 구독 관리 (`/admin/users/{id}/subscriptions/{app}`)
5. 감사 로그 조회 (`/admin/audit/search` 프론트)

참조 문서: `../docs/api-spec.md`(API 계약), `../docs/security-model.md`(인증/capability 모델), `../docs/prd.md`(아키텍처).

## 2. 기술 스택

- **Go 서버 렌더링**: `html/template` + htmx (필요 시). SPA/별도 빌드 파이프라인 금지.
- 기존 모노레포 스택 재사용: chi 라우터, pgx, 루트 `go.mod` 공유. 새 바이너리는 `cmd/adminui/`.
- 관리 API는 가능하면 **기존 `/admin` HTTP API를 호출**해 재사용하고, API가 없는 작업(키 발급/폐기)만 DB 직접 접근한다. 비즈니스 로직을 UI에 중복 구현하지 않는다.
- 스타일: 외부 CDN 금지(오프라인 동작), 정적 자산은 `embed.FS`로 바이너리에 포함.

## 3. 보안 경계 (타협 불가)

- 관리자 UI는 **공개 인터넷에 노출하지 않는다** — Tailscale/사설망 전제. 리슨 기본값 `127.0.0.1`.
- UI 자체 로그인 세션과 서버 API 키를 분리한다. UI가 사용하는 admin API 키는 서버 측 설정으로만 주입(브라우저에 절대 전달 금지).
- 발급된 평문 API 키는 **응답 1회 렌더링 후 어디에도 저장/로그하지 않는다**. 로그·감사 기록에는 prefix만.
- CSRF 보호 필수(상태 변경 POST 전부). 모든 관리 행위는 기존 audit chain에 기록되도록 `/admin` API 경유를 우선한다.
- 루트 R7 준수: secret은 `.env`/환경변수로만, placeholder 기본값 유지.

## 4. 작업 원칙 (Claude Fable 5 가이드 기반)

### 4.1 충분하면 실행한다

정보가 충분하면 즉시 실행한다. 대화에서 이미 확정된 사실을 다시 도출하거나, 사용자가 이미 내린 결정을 재론하거나, 추구하지 않을 옵션을 사용자 대면 메시지에서 나열하지 않는다. 선택지를 저울질할 때는 전수 조사가 아니라 **추천안 하나**를 제시한다.

### 4.2 범위 규율

작업에 필요한 것 이상의 기능 추가·리팩토링·추상화를 하지 않는다. 버그 수정에 주변 정리를 끼워 넣지 않고, 일회성 작업에 헬퍼를 만들지 않는다. 가상의 미래 요구사항을 위해 설계하지 않는다 — 잘 동작하는 가장 단순한 것을 만든다. 발생할 수 없는 시나리오에 대한 에러 처리·폴백·검증을 추가하지 않는다. 내부 코드와 프레임워크 보장은 신뢰하고, 검증은 시스템 경계(사용자 입력, 외부 API)에서만 한다. 코드를 그냥 바꾸면 되는 곳에 feature flag나 하위 호환 shim을 넣지 않는다.

### 4.3 결과 우선 보고

작업 완료 후 첫 문장은 "무슨 일이 있었나 / 무엇을 발견했나"에 답한다 — 사용자가 "TLDR만 달라"고 했을 때 줄 내용. 근거와 상세는 그 뒤에. 짧게 쓰는 방법은 문장을 조각·약어·화살표 체인(`A → B → 실패`)으로 압축하는 게 아니라 **무엇을 포함할지 선별**하는 것이다(독자의 다음 행동을 바꾸지 않는 세부는 삭제). 가독성이 간결성보다 우선한다.

### 4.4 체크포인트 — 멈춰야 할 때만 멈춘다

다음 경우에만 사용자에게 묻고 턴을 끝낸다:
- 파괴적이거나 되돌리기 어려운 작업
- 실질적 범위 변경
- 사용자만 제공할 수 있는 입력

이외에는 진행한다. 약속("이제 X를 하겠습니다")으로 턴을 끝내지 않는다 — 마지막 문단이 계획·질문·다음 단계 목록·미수행 작업에 대한 약속이라면, 지금 도구 호출로 그 작업을 한다.

### 4.5 증거 기반 진행 보고

진행 상황을 보고하기 전에 각 주장을 이 세션의 도구 결과와 대조한다. 증거를 가리킬 수 있는 작업만 보고하고, 미검증이면 명시한다. 테스트가 실패하면 출력과 함께 실패라고 말하고, 건너뛴 단계는 건너뛰었다고 말하고, 완료·검증된 것은 hedging 없이 명료하게 말한다.

### 4.6 경계 — 진단과 수정을 구분한다

사용자가 문제를 설명하거나 질문하거나 생각을 정리하는 중이라면, 산출물은 **평가/분석**이다. 발견을 보고하고 멈춘다 — 수정 요청이 오기 전에 고치지 않는다. 시스템 상태를 바꾸는 명령(재시작, 삭제, 설정 변경) 전에는 증거가 그 행동을 실제로 뒷받침하는지 확인한다. 알려진 장애와 패턴이 비슷한 신호라도 원인은 다를 수 있다.

### 4.7 서브에이전트 위임

독립적인 하위 작업은 서브에이전트에 위임하고, 그들이 도는 동안 계속 작업한다. 서브에이전트가 궤도를 벗어나거나 필요한 컨텍스트가 없으면 개입한다. 루트 R4의 OMC 라우팅(executor/code-reviewer/security-reviewer/verifier)을 따르되, **복잡한 작업(설계·보안 로직·대규모 구현·심층 리뷰)은 `model: "fable"`(Fable 5)로 스폰**한다 (사용자 지시, R4의 opus 문구보다 우선). 단순 기계적 작업은 상속/sonnet. 장기 실행 작업에서는 **깨끗한 컨텍스트의 검증자 서브에이전트**가 자기 비평보다 낫다 — 단계 완료마다 명세 대비 검증을 서브에이전트로 돌린다.

### 4.8 교훈 기록

이전 실행에서 얻은 교훈은 `.omc/notepad.md`(루트 규약)에 기록한다: 교훈 하나당 항목 하나, 첫 줄에 한 줄 요약. 정정과 확인된 접근법 모두, 이유와 함께. 레포/대화 이력이 이미 기록하는 것은 저장하지 않고, 중복 생성 대신 기존 항목을 갱신하며, 틀린 것으로 판명된 노트는 삭제한다.

### 4.9 자율 실행 모드

phase 자동 진행(루트 R3) 중에는 사용자가 실시간으로 보고 있지 않다. "~할까요?"는 작업을 블로킹한다 — 원 요청에서 따라 나오는 가역적 작업은 묻지 않고 진행한다. 컨텍스트 잔량을 이유로 작업을 멈추거나 요약·새 세션을 제안하지 않는다.

### 4.10 요청에 이유를 담는다

서브에이전트에 위임하거나 문서를 작성할 때, 무엇을 할지만이 아니라 **누구를 위한 것이고 그 출력이 무엇을 가능하게 하는지**를 함께 전달한다. (예: "운영자가 브라우저에서 키를 발급하기 위한 폼 — 평문은 1회만 보여야 함")

## 5. 검증 파이프라인 (루트 R2 구체화)

phase 완료 보고 전 순서대로:
1. `go build ./... && go vet ./... && go test ./...`
2. live smoke: `docker compose` 스택 위에서 UI 주요 흐름 실제 실행 (키 발급 → 발급 키로 API 호출 성공까지)
3. 보안 확인: 평문 키 로그/저장 없음, CSRF 토큰 동작, 리슨 주소 확인
4. commit & push (`.omc/state/` churn 제외)
5. `docs/phase-reports/` 보고서 + README 인덱스 갱신
